# Audit Report — 2026-06-03

Target: gestion-poker operator (Gestor de Mesas) — primary surface, plus SeatModal and PlayerCard.

## Anti-Patterns Verdict

**Pass, with caveats.** The codebase is operationally serious (no marketing fluff, no hero metrics, no identical card grids at the page level). The two `seats` of AI slop are (1) the 95-line inline `<style>` block in `player-card.js` and (2) the `iOS blue/green/amber/red` palette that doesn't match the gold-on-black system. The operator is denser than typical product UI; the lobby carries the glass register. Reads as built, not generated.

## Detailed Findings by Severity

### P0 — Blocking

#### [P0] XSS in waiting-list render
- **Location**: `src/components/waiting-list/waiting-list.js:117-125`
- **Category**: Security / Accessibility
- **Impact**: Player name from Firestore is injected via `innerHTML` template literal. A player with `<img src=x onerror=alert(1)>` in their name would execute arbitrary JS in the staff's browser, which has access to the Firebase Admin SDK and full table data.
- **WCAG/Standard**: OWASP A03:2021 (Injection)
- **Recommendation**: Build with `textContent` / DOM API, or escape via a small helper. Use `data-id` only, render name with `el.textContent = item.name` after element creation.
- **Suggested command**: `$impeccable harden`

#### [P0] No `error` fallback in player-card onSnapshot
- **Location**: `src/components/player-card/player-card.js:562-564`
- **Category**: Reliability
- **Impact**: `console.error` on snapshot error is the only signal; UI never tells the operator that player state is stale. Critical under sala noise.
- **WCAG/Standard**: WCAG 4.1.3 (Status Messages)
- **Recommendation**: Surface a non-blocking error toast (aria-live=polite) when snapshot fails; retry once; then require manual reload.
- **Suggested command**: `$impeccable harden`

#### [P0] `getTableReportData()` returns `null` placeholder
- **Location**: `src/components/side-panel/side-panel.js:473`
- **Category**: Functional
- **Impact**: CSV export button (if reachable) opens an alert then bails. Feature is half-built; `handleExportCSV` is wired but the function it calls returns null unconditionally.
- **WCAG/Standard**: WCAG 2.1 AA (functional completeness)
- **Recommendation**: Either implement the function against `getTableReportData` aggregation pattern (rake, propinas, jackpot, RPH) or remove the button until implemented.
- **Suggested command**: `$impeccable audit` (re-scope) or `$impeccable craft`

#### [P0] `export-csv-btn` not in HTML template
- **Location**: `src/components/side-panel/side-panel.js:282` (refs) and template at `side-panel.js:44-96` (no `#export-csv-btn` defined)
- **Category**: Functional
- **Impact**: `document.getElementById('export-csv-btn')` returns `null`; subsequent `addEventListener` is a no-op. But the function `handleExportCSV` is never bound to any element, so clicking "Export CSV" (if user could find it) would do nothing. The button isn't actually rendered.
- **WCAG/Standard**: —
- **Recommendation**: Add the button to the panel-header__actions next to the mirror button (with icon), or remove the handler.
- **Suggested command**: `$impeccable audit`

### P1 — Major

#### [P1] 95-line inline `<style>` block in player-card.js
- **Location**: `src/components/player-card/player-card.js:128-222`
- **Category**: Theming / Anti-Pattern / Code Quality
- **Impact**: Bypasses design system entirely. The SCSS file at `player-card.scss` has a parallel set of styles that mostly don't match. Adding a token requires editing two places. CSS injected via JS can't be tree-shaken, hot-reloaded, or audited.
- **WCAG/Standard**: Maintainability, single source of truth
- **Recommendation**: Move all inline styles to `player-card.scss`. Use existing tokens (`$brand-gold`, `$text-light`, `$glass-border`). Drop the `<style>` tag from the template; rely on external CSS.
- **Suggested command**: `$impeccable typeset` + `$impeccable audit`

#### [P1] iOS palette drift (#30d158, #5ac8fa, #ff453a)
- **Location**: `src/components/player-card/player-card.js:172, 184, 186, 194, 220, 588-612, 706`; `poker-table.scss` `absencePulse` (`#ff453a`)
- **Category**: Theming
- **Impact**: iOS blue/green/red palette does not exist in DESIGN.md. The design system prescribes `#38b000` (state-ok), `#FFD54F` (state-money), `#FFD60A` (warning), `#e63946` (danger). Three palettes fighting for the same screen.
- **WCAG/Standard**: Visual consistency
- **Recommendation**: Replace all `#30d158` → `#38b000`, `#5ac8fa` → `#d9b66f` (gold for emphasis), `#ff453a` → `#e63946` (state-danger, with urgent kept distinct via animation speed not color). The iOS green is *almost* the same as state-ok; just collapse.
- **Suggested command**: `$impeccable colorize` + `$impeccable audit`

#### [P1] Waiting-list query re-created on every table change
- **Location**: `src/components/waiting-list/waiting-list.js:79-97`
- **Category**: Performance
- **Impact**: `onSnapshot(tableRef, ...)` fires on every table doc update — including the 1Hz session timer updates. Each fire re-creates the `wlUnsub`, runs a new `query(...where, where, where, orderBy)`, and re-subscribes. Reads = writes per second.
- **WCAG/Standard**: Performance
- **Recommendation**: Hoist the `gameType/blinds` reading out of the snapshot callback. Read once at init, or only react to `gameType/blinds` field changes (not the whole doc).
- **Suggested command**: `$impeccable optimize`

#### [P1] Two parallel waiting-list naming systems
- **Location**: `src/components/waiting-list/waiting-list.js:117-125` uses `.wl-item-new`, `.wl-item-num`, `.wl-item-dot`, `.wl-item-name`, `.wl-item-sit`, `.wl-item-remove`. `src/components/waiting-list/waiting-list.scss:58-150` styles `.waiting-list-item`, `.wl-avatar`, `.wl-name`, `.wl-action-btn`. The two are not the same.
- **Category**: Code Quality / Design System drift
- **Impact**: SCSS in the project is dead — none of the styled classes are used by the rendered HTML. Visual treatment of the waiting list is whatever the browser default is.
- **WCAG/Standard**: Consistency
- **Recommendation**: Pick one vocabulary. Given the rest of the codebase uses `*-new` (`wl-add-btn-new`, `wl-input-new`, `wl-submit-new`) consistently in `side-panel.js`, change SCSS to match the actual rendered HTML.
- **Suggested command**: `$impeccable audit`

#### [P1] Inline styles in side-panel template
- **Location**: `src/components/side-panel/side-panel.js:81-82` (`style="margin:0"`, `style="display:flex;align-items:center;gap:8px"`)
- **Category**: Code Quality / Theming
- **Impact**: Visual rules stuck inside a JS string template. No theme switch, no responsive override, no design-token use.
- **WCAG/Standard**: Maintainability
- **Recommendation**: Extract to a class (`.wl-header__actions`) and put in `side-panel.scss`.
- **Suggested command**: `$impeccable layout`

#### [P1] `seat-modal.jsx` is React but the surrounding ecosystem is vanilla JS
- **Location**: `src/components/seat-modal/SeatModal.jsx` is mounted via `react-dom/client` from `Lobby.jsx` / `tables-manager.js`; `side-panel.js`, `waiting-list.js`, `player-card.js` are vanilla.
- **Category**: Architecture (not strictly a defect, but a drift signal)
- **Impact**: Two worlds; bundle splits, but rendering pipeline is split too. Mental model is harder for new contributors.
- **WCAG/Standard**: —
- **Recommendation**: Accept the split (it's the project's chosen pattern) but document the rule. The `React.isOpen`-based portal pattern in SeatModal is clean; don't fragment further.
- **Suggested command**: `$impeccable document`

#### [P1] Inline styles in SeatModal
- **Location**: `src/components/seat-modal/SeatModal.jsx:262, 365, 369-380, 385-405, 397`
- **Category**: Theming / Code Quality
- **Impact**: ~8 inline `style={}` blocks. Spacing, typography, transitions inline. Cannot theme or audit.
- **WCAG/Standard**: —
- **Recommendation**: Move to `seat-modal.scss`. Add classes for `.sm-avatar-toggle`, `.sm-avatar-chevron`, `.sm-avatar-grid`, `.sm-avatar-list`.
- **Suggested command**: `$impeccable typeset`

### P2 — Minor

#### [P2] `Math.random`-like pattern in `subscribeSeatRealtime`: status check is over-broad
- **Location**: `src/components/player-card/player-card.js:497` — `isOccupiedDoc = (status === 'occupied') || chips > 0 || hasName`
- **Category**: Logic
- **Impact**: A seat with `name` set but `chips=0` and `status='empty'` is considered occupied. Could mislead the operator.
- **Recommendation**: Prefer status check; only fall back to chips/name if status is missing.

#### [P2] No cleanup for `sessionInterval` and `_liveTimer` on unmount
- **Location**: `src/components/side-panel/side-panel.js:36, 167-177`; `src/components/player-card/player-card.js:47, 400-409, 599`
- **Category**: Performance / Memory
- **Impact**: Intervals persist if the module is re-imported. In a SPA this is rare, but in display/mirror pages where `initSidePanel` may be called multiple times, leak accumulates.
- **Recommendation**: Return an `unmount()` function that clears all intervals and unsubscribes all `onSnapshot`s.

#### [P2] `onSnapshot` in `side-panel.js` re-renders timer on every table update
- **Location**: `src/components/side-panel/side-panel.js:410-415` and `174` (the 1s ticker)
- **Category**: Performance
- **Impact**: Two timers, one driven by Firestore updates (irregular), one by 1s interval. The Firestore one is redundant — if table doesn't update, only the interval ticks. If table updates, both fire.
- **Recommendation**: Drop the Firestore-driven render for the timer; let the 1s interval be the source of truth.

#### [P2] `alert()` used for user-facing errors
- **Location**: `src/components/waiting-list/waiting-list.js:48`; `src/components/player-card/player-card.js:629, 677, 706`; `src/components/side-panel/side-panel.js:440, 443`
- **Category**: UX / Accessibility
- **Impact**: Native browser alert blocks the thread, breaks immersive UI, and isn't keyboard-accessible the way the rest of the modal is. Also can't be styled.
- **Recommendation**: Use a toast system (`react-hot-toast` is already in the project via lucide-react ecosystem? Check) or build a minimal `useToast` hook. Place in design system.

#### [P2] `unnecessary` re-renders on every input keystroke in SeatModal

## Patterns & Systemic Issues

1. **Theming bypass in player-card.js** is the single largest design-system drift in the codebase. 95 lines of inline `<style>` reimplement tokens that exist in `_tokens.scss`, `_variables.scss`, and the new `DESIGN.md`. **Fix this before adding new components**, otherwise the same pattern propagates.

2. **Module-level state (timerRoot, currentSeatId, sessionInterval, etc.) is the dominant pattern in vanilla JS components**. This is fine for a single-page app, but assumes a single instance. If display mirrors run side-by-side, the timerRoot will fight between panels.

3. **Interval cleanup is rare**. `sessionInterval`, `_liveTimer`, `_absenceTimer`, `startPanelTicker` — none of them are cleared on unmount. In a PWA this matters because the service worker can keep the page alive across sessions.

4. **No `aria-live` regions anywhere**, despite the brand promising "errores visuales + auditivos. Errores críticos deben anunciarse". `alert()` is the only "audible" signal and it's blocking the wrong way.

## Positive Findings

- Operator surfaces are dense without being maximalist — good information density.
- Glassmorphism is contained to the lobby and card surfaces; the operator reads as a terminal, not a SaaS template.
- Timer is the visual anchor of session state; it correctly uses `font-variant-numeric: tabular-nums`.
- `prefers-reduced-motion` is respected in `poker-table.scss` (sheen, flip) and `waiting-list.scss`. Not universal, but a strong start.
- Status states (Ausente / Ausente URGENTE / En mesa) are present in player-card; redundancy with text+color is in place.
- Firestore batched writes in `pauseAllSeatTimers` and `resumeAllSeatTimers` show transactional awareness.
